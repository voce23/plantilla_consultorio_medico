Lección 10.6: Deploy a Producción

Checklist: env, migrate, storage:link, optimize

🎯 Piensa en esto: Antes de abrir una nueva sucursal de tu clínica, tienes que revisar una lista: ¿está el personal listo? ¿está el equipo médico instalado? ¿están los seguros al día? ¿tiene luz y agua? El deploy a producción es ese checklist final antes de "abrir las puertas" de tu aplicación al mundo real.
1 Configurar el .env de producción

El archivo .env de producción es diferente al de desarrollo. Los cambios críticos:

📍 Archivo: .env (en el servidor de producción)

# ⚠️ CAMBIOS CRÍTICOS para producción

# 1. Modo producción (NUNCA dejar debug en true)
APP_ENV=production
APP_DEBUG=false

# 2. URL correcta del dominio
APP_URL=https://clinicasalud.com

# 3. Base de datos de producción (no la local)
DB_HOST=tu-servidor-db.com
DB_DATABASE=clinica_produccion
DB_USERNAME=usuario_produccion
DB_PASSWORD=password_seguro_largo

# 4. Cache en Redis (más rápido que filesystem)
CACHE_DRIVER=redis
SESSION_DRIVER=redis
QUEUE_CONNECTION=redis

💡 APP_DEBUG=false es lo más importante. Con debug en true, los errores muestran contraseñas y rutas del servidor.

📝 ESTE ES EL CÓDIGO QUE VAMOS A APRENDER:

📍 Terminal del servidor de producción

# El deploy completo en orden correcto
php artisan down --message="Actualizando sistema, volvemos en 5 minutos"

git pull origin main

composer install --no-dev --optimize-autoloader

npm ci ; npm run build

php artisan migrate --force

php artisan storage:link

php artisan config:cache
php artisan route:cache
php artisan view:cache

php artisan filament:upgrade
php artisan filament:cache-components

php artisan up

🔍 Vamos a entender línea por línea (como si fuera la primera vez):

php artisan down --message="..."
→ Pone la aplicación en modo mantenimiento. Los usuarios ven una página "En mantenimiento" mientras haces cambios. Sin esto, podrían ver errores.
composer install --no-dev --optimize-autoloader
→ --no-dev no instala paquetes de desarrollo (testing, debugbar, etc). --optimize-autoloader genera un autoloader más rápido.
php artisan migrate --force
→ --force es necesario en producción porque Laravel pide confirmación antes de migrar. Sin --force, el deploy fallaría en scripts automáticos.
php artisan storage:link
→ Crea un enlace simbólico de public/storage a storage/app/public. Sin esto, las imágenes subidas no serán accesibles desde el navegador.
php artisan up
→ Quita el modo mantenimiento. La aplicación vuelve a estar en línea para los usuarios.
2 Permisos de carpetas

Las carpetas storage y bootstrap/cache necesitan permisos de escritura:

# En servidor Linux
chmod -R 775 storage bootstrap/cache
chown -R www-data:www-data storage bootstrap/cache

# Verificar que las carpetas existen
mkdir -p storage/app/public
mkdir -p storage/framework/cache
mkdir -p storage/framework/sessions
mkdir -p storage/framework/views
mkdir -p storage/logs

# En cPanel o hosting compartido
chmod -R 755 storage
chmod -R 755 bootstrap/cache

💡 Si no se configuran los permisos, Laravel no puede escribir logs ni guardar sesiones y el sitio fallará.

3 Checklist de seguridad

Antes de abrir el sitio al público, verificar:

# ✅ Verificar variables críticas
grep "APP_DEBUG" .env          # Debe ser false
grep "APP_ENV" .env            # Debe ser production

# ✅ Verificar que .env NO está en el repositorio
cat .gitignore | grep ".env"   # Debe aparecer

# ✅ Verificar HTTPS
# En Nginx: escuchar en puerto 443
# En Apache: habilitar SSL

# ✅ Generar APP_KEY si es nuevo servidor
php artisan key:generate

# ✅ Headers de seguridad en Nginx
# add_header X-Content-Type-Options nosniff;
# add_header X-Frame-Options SAMEORIGIN;
# add_header X-XSS-Protection "1; mode=block";

💡 Nunca subas el .env a Git. Contiene contraseñas y la APP_KEY que cifra las sesiones.

4 Configuración del servidor web (Nginx)

Configuración básica de Nginx para Laravel:

server {
    listen 443 ssl;
    server_name clinicasalud.com www.clinicasalud.com;
    root /var/www/clinica/public;

    index index.php;

    # Todas las rutas pasan por index.php (necesario para Laravel)
    location / {
        try_files $uri $uri/ /index.php?$query_string;
    }

    # Ejecutar PHP
    location ~ \.php$ {
        fastcgi_pass unix:/var/run/php/php8.2-fpm.sock;
        fastcgi_param SCRIPT_FILENAME $realpath_root$fastcgi_script_name;
        include fastcgi_params;
    }

    # No mostrar .env ni archivos ocultos
    location ~ /\.(?!well-known).* {
        deny all;
    }

    # SSL
    ssl_certificate /etc/letsencrypt/live/clinicasalud.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/clinicasalud.com/privkey.pem;
}

💡 try_files $uri $uri/ /index.php es la línea clave para que Laravel maneje todas las rutas.

5 Checklist final de verificación

Después del deploy, probar estas funcionalidades:

✅ FRONTEND
   □ Página de inicio carga
   □ Imágenes se muestran
   □ Formulario de citas funciona
   □ Google Maps se muestra
   □ Links de redes sociales abren
   □ Política de privacidad accesible

✅ ADMINISTRACIÓN  
   □ /admin carga
   □ Login funciona
   □ Listado de registros funciona
   □ Crear/Editar/Eliminar funciona
   □ Subida de imágenes funciona

✅ PERFORMANCE
   □ Google PageSpeed > 80 puntos
   □ Sin errores en consola del navegador
   □ HTTPS activo (candado verde)

✅ SEO
   □ Meta tags presentes (ver código fuente)
   □ Schema.org válido (Google Rich Results)
   □ Sitemap en /sitemap.xml

💡 Google PageSpeed Insights: https://pagespeed.web.dev — gratis y muy útil.

6 Rollback de emergencia

Si algo sale mal después del deploy:

# Poner en mantenimiento inmediatamente
php artisan down --message="Realizando mantenimiento urgente"

# Volver al commit anterior en Git
git log --oneline -5    # Ver commits recientes
git reset --hard HEAD~1 # Volver al commit anterior

# Revertir migraciones si es necesario
php artisan migrate:rollback  # Revierte la última migración

# Restaurar backup de base de datos
mysql -u usuario -p base_de_datos < backup-antes-deploy.sql

# Limpiar cachés
php artisan optimize:clear

# Levantar el sitio
php artisan up

💡 Siempre haz un backup de la base de datos ANTES de cada deploy. Es el seguro de vida de tu aplicación.
