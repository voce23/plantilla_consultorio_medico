Lección 10.3: Performance

Lazy loading imágenes, defer scripts, preconnect fonts

🎯 Piensa en esto: Imagina que pides comida a domicilio. No es eficiente que el repartidor te traiga todo el restaurante junto. Lo ideal es que traiga primero lo que vas a comer de inmediato, y el postre después. La optimización de performance funciona igual: cargamos primero lo que el usuario necesita ver ahora, y lo demás lo dejamos para después.
1 Preconnect: conectarse antes de pedir

Le dice al navegador que establezca la conexión con servidores externos desde el inicio:

📍 Archivo: resources/views/layouts/app.blade.php

<head>
    {{-- Preconnect para Google Fonts (performance crítica) --}}
    <link rel="preconnect" href="https://fonts.googleapis.com">
    <link rel="preconnect" href="https://fonts.gstatic.com" crossorigin>
    
    {{-- DNS prefetch para recursos menos críticos --}}
    <link rel="dns-prefetch" href="https://fonts.googleapis.com">
    <link rel="dns-prefetch" href="https://www.google-analytics.com">
    
    {{-- Ahora sí cargar la fuente --}}
    <link href="https://fonts.googleapis.com/css2?family=Inter:wght@400;600&display=swap" 
          rel="stylesheet">
</head>

💡 preconnect establece la conexión TCP+DNS antes de necesitarla. Puede ahorrar 100-300ms.

📝 ESTE ES EL CÓDIGO QUE VAMOS A APRENDER:

📍 Archivo: Cualquier vista Blade con imágenes

<!-- Imagen con lazy loading nativo -->
<img 
    src="{{ asset('images/doctor.jpg') }}"
    alt="Dr. García - Cardiólogo"
    loading="lazy"
    decoding="async"
    class="w-full h-48 object-cover"
>

<!-- Primera imagen (hero): carga inmediata con prioridad alta -->
<img 
    src="{{ asset('images/hero.jpg') }}"
    alt="Clínica Salud - Atención de calidad"
    loading="eager"
    fetchpriority="high"
    class="w-full object-cover"
>

🔍 Vamos a entender línea por línea (como si fuera la primera vez):

loading="lazy"
→ El navegador solo descarga la imagen cuando está a punto de entrar al viewport. Las imágenes fuera de pantalla se saltan hasta que el usuario hace scroll.
decoding="async"
→ El navegador puede decodificar la imagen en segundo plano, sin bloquear la renderización del resto de la página.
fetchpriority="high"
→ Indica que esta imagen es crítica (como el hero). El navegador la descarga con máxima prioridad. Solo úsala en la imagen principal.
loading="eager"
→ Carga inmediata. Úsala para el hero o imágenes visibles sin hacer scroll. Es el comportamiento por defecto.
2 Scripts con defer

Los scripts con defer se descargan en paralelo pero se ejecutan después de que el HTML esté listo:

<head>
    {{-- Scripts de análisis con defer (no bloquean la carga) --}}
    <script defer src="https://analytics.example.com/tracker.js"></script>
    
    {{-- Scripts externos con async (se ejecutan cuando estén listos) --}}
    <script async src="https://www.googletagmanager.com/gtag/js"></script>
    
    {{-- Vite compila automáticamente con module scripts (equivale a defer) --}}
    @vite(['resources/css/app.css', 'resources/js/app.js'])
</head>

💡 defer: respeta el orden. async: se ejecuta en cuanto descarga. Vite ya usa module que incluye defer.

3 Componente responsive-image de Blade

El componente que creamos genera automáticamente los atributos correctos:

📍 Archivo: resources/views/components/responsive-image.blade.php

<!-- Imagen de doctor con fade-in al cargar -->
<x-responsive-image 
    src="{{ asset('storage/' . $doctor->foto) }}"
    alt="Dr. {{ $doctor->nombre }} - {{ $doctor->especialidad }}"
    class="w-full h-64 object-cover rounded-xl"
    loading="lazy"
/>

<!-- Hero image con prioridad alta -->
<x-responsive-image 
    src="{{ asset('images/hero-clinica.jpg') }}"
    alt="Clínica Salud - Tu salud es nuestra prioridad"
    class="w-full h-[600px] object-cover"
    loading="eager"
    fetchpriority="high"
/>

💡 El componente añade automáticamente el efecto fade-in cuando la imagen termina de cargar.

4 Cachés de Laravel para producción

En producción, Laravel puede cachear rutas, vistas y configuración:

📍 Terminal (se ejecuta en el servidor)

# Cachear todo para máxima velocidad
php artisan config:cache   # Agrupa config/*.php en un archivo
php artisan route:cache    # Precalcula todas las rutas
php artisan view:cache     # Precompila todas las vistas Blade
php artisan event:cache    # Cachea los event listeners

# Ver el efecto: antes y después de cachear
php artisan route:list     # Lista todas las rutas

# Para desarrollo, limpiar cachés
php artisan optimize:clear

💡 optimize:clear limpia todo. Úsalo cuando hagas cambios en desarrollo y algo no se actualice.

5 Query optimization: evitar N+1

Un error común que hace el sitio lento: hacer consultas dentro de un loop:

// ❌ MALO: N+1 consultas (1 por cada doctor)
$doctores = Doctor::all();
@foreach($doctores as $doctor)
    {{ $doctor->especialidad->nombre }} {{-- Consulta por cada doctor --}}
@endforeach

// ✅ BUENO: 2 consultas en total con eager loading
$doctores = Doctor::with('especialidad')->get();
@foreach($doctores as $doctor)
    {{ $doctor->especialidad->nombre }} {{-- Ya cargado, 0 consultas extra --}}
@endforeach

💡 with('relacion') carga la relación en una sola consulta adicional, no una por cada fila.

6 Formatos de imagen modernos

Usar WebP reduce el tamaño de las imágenes hasta un 30%:

<!-- Usar picture para ofrecer WebP con fallback -->
<picture>
    <source 
        srcset="{{ asset('images/doctor.webp') }}" 
        type="image/webp">
    <img 
        src="{{ asset('images/doctor.jpg') }}" 
        alt="Dr. García"
        loading="lazy"
        class="w-full h-48 object-cover">
</picture>

💡 El navegador usa WebP si lo soporta, y cae al JPG si no. WebP es 25-34% más ligero.

🎉 ¿Qué aprendimos hoy?

  • preconnect establece conexiones anticipadas con servidores externos
  • loading="lazy" retrasa imágenes fuera de pantalla
  • fetchpriority="high" prioriza la imagen hero
  • defer en scripts no bloquea la renderización
  • php artisan config:cache optimiza para producción
  • with('relacion') evita el problema N+1
  • WebP reduce el peso de las imágenes hasta un 34%
